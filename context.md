EvidentMed — Website Positioning & Messaging Brief

Purpose of This Document

This document briefs the product, design, and marketing teams on the core positioning, narrative, and architectural principles behind EvidentMed. It should guide website structure, messaging hierarchy, visual direction, and copy tone.

Naming Convention for Website Copy

Use `Evident` in all on-page copy, headings, metadata, nav labels, and UI text.
Use `EvidentMed` only in the logo image asset name (`evidentmed.png`) and not as visible written brand copy on the site.

⸻

1. Master Brand Context

Evident is a platform that transforms unstructured real-world clinical data into structured, computable intelligence.
	•	Evident Vet → Veterinary clinical data intelligence
	•	EvidentMed → Human clinical trial eligibility intelligence

EvidentMed is not a standalone “trial matching tool.” It is the human-medicine application of Evident’s core knowledge graph and computation engine.

⸻

2. Core Positioning

One-Line Positioning

EvidentMed transforms unstructured hospital data into computable clinical intelligence — enabling automated, local-first trial eligibility screening at site and network scale.

Category

Federated Clinical Eligibility Infrastructure

Identity

EvidentMed is:
	•	A clinical knowledge graph engine
	•	An automated eligibility computation system
	•	A distributed research orchestration layer
	•	A local-first infrastructure platform

EvidentMed is NOT:
	•	A patient recruitment marketplace
	•	A centralised patient registry
	•	A data broker
	•	A marketing-based recruitment tool

We are infrastructure.

⸻

3. The Core Problem

Clinical trial recruitment is fundamentally a data problem.
	•	Eligibility criteria are written in natural language
	•	Patient data lives in unstructured clinical notes
	•	Screening is manual and resource-intensive
	•	Feasibility assessments are inaccurate or slow
	•	Sponsors lack real-time visibility into network capacity
	•	Sites are overwhelmed by chart review

Trials are delayed not because of science — but because eligibility is not computable at scale.

⸻

4. The Core Solution

EvidentMed builds a computable layer on top of hospital data.

Step 1 — Ingestion

Structured and unstructured patient data is ingested locally at each site.

Step 2 — Knowledge Graph Construction

Clinical records are transformed into dynamic, structured knowledge graphs.

Step 3 — Eligibility Logic Translation

Trial eligibility criteria (written in prose) are translated into computable logic.

Step 4 — Automated Computation

Every patient can be continuously evaluated against every active trial.

Result:
	•	Real-time eligibility detection
	•	Automated screening
	•	Instant feasibility estimation
	•	Reduced manual chart review

⸻

5. Architectural Principle: Local-First & Federated

This is a core differentiator and must be clearly reflected in website messaging.

Key Principle

Computation happens where the data lives.

What This Means
	•	All data processing can run locally at each hospital node.
	•	Raw patient data does not need to leave the site.
	•	Sites retain full control and ownership of their data.
	•	Only aggregate, non-identifiable signals are shared across networks.
	•	Eligibility logic can be distributed across sites without centralising patient records.

This enables:
	•	GDPR-aligned deployment
	•	Privacy-by-design architecture
	•	Scalable multi-site orchestration

Core conceptual framing:
We don’t move data. We move computation.

	•	Not overly promotional
	•	Not AI buzzword-heavy

	•	Node-based architecture visuals
	•	Knowledge graph abstractions
